# AGENTS.md

Работай по-русски, структурно и по делу.

## Перед началом

1. Сначала прочитай этот файл.
2. Затем прочитай `docs/AI_INDEX.md`, если он есть.
3. Для общего контекста используй `README.md` и релевантные разделы `docs/AI_INDEX.md`; отдельный `context.md` больше не ведётся.
4. Если задача зависит от актуального API библиотек или сервисов, используй Context7 MCP. Если Context7 недоступен, используй официальную документацию.
5. Не сканируй весь проект без причины: сначала определи релевантные файлы по задаче.

## Коммиты

- Делай коммиты после законченного изменения.
- Не делай push, если пользователь прямо не попросил.
- В грязном worktree не трогай чужие изменения. Стадь и коммить только свои файлы.

## Архитектурные правила

- Транспорт, UI и GUI должны быть тонкими.
- Бизнес-логика должна жить в сервисах.
- Работа с файлами, SQL и другими хранилищами должна быть вынесена в repositories/data layer.
- Не смешивай UI-логику с domain-логикой.
- Для этого проекта главный legacy-flow сейчас находится в ноутбуке `MPstats_export_yearmonnth_v1.3_pipeline.ipynb`; при доработках выноси новую переиспользуемую логику в Python-модули рядом с `pipeline/` или `classifiers/`.
- Для web-app не заставляй пользователя редактировать сырой JSON/CSV, если можно дать нормальную форму. CRUD правил и справочников держи в services/data layer, routes оставляй тонкими.

## Источники правды

- Основной пайплайн: `MPstats_export_yearmonnth_v1.3_pipeline.ipynb`.
- Индекс для агентов: `docs/AI_INDEX.md`.
- Человеческий вход в проект: `README.md`.
- Пользовательская инструкция: `docs/USER_GUIDE.md`.
- Справочник категорий: `Справочник категорий MP STATS.csv`.
- Синтаксис фильтров MPStats: `filter.md`.
- Архив задач: `справочник tasks архив.md`.

- Локальное приложение: `mpstats_app/` + `web/`; справочник и правила редактируются через web UI и сохраняются обратно в CSV.

## Изменения

- Делай минимальный локальный diff.
- Не трогай unrelated flows.
- Не меняй analytics/event names или имена выходных колонок без явного требования.
- Если меняется schema, SQL, repository или model, опиши DB impact и добавь/обнови миграцию и тест.
- Не предлагай заменять production-БД локальной.
- При работе с `TASKS` копируй значения из справочника категорий буквально: пробелы, регистр, фильтры и path не нормализуй без запроса.
- Для справочника категорий используй только CSV `Справочник категорий MP STATS.csv`; Excel-дубликаты не считай источником правды.
- При любой правке, которая меняет пользовательский сценарий, интерфейс, расчёты, статусы, структуру результатов, справочник, классификатор или workflow, обновляй `docs/USER_GUIDE.md` в том же изменении.

## Проверки

- Запускай самый узкий релевантный тест или проверку.
- Для классификатора проверяй импорт/compile модулей `classifiers`.
- Для конфигов шага 1 проверяй импорт/compile модулей `pipeline`.
- Для notebook-логики указывай, какие шаги ноутбука надо перезапустить вручную.
- Если тесты не запускались, честно напиши почему и что запустить вручную.

Всегда укажи:

- изменённые файлы;

- запущенные проверки;
- что проверить вручную.
